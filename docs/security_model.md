# RMLW Security Model

## Threat Model

### Intended Use

RMLW is designed for **authorised security testing** in controlled environments:

- Lab networks
- Staging environments
- Applications you own or have explicit permission to test

### Out of Scope

- Unauthorised scanning of third-party systems
- Production systems without approval
- Any use that violates computer misuse laws

## Tool Design Principles

### Transparency

RMLW is not a black-box scanner:

- Every finding records the exact payload and response characteristics
- Logs show which tests ran and what was observed
- Learning components (RL, evolution) are inspectable

### Least Privilege

- The scanner does not execute arbitrary system commands
- No dynamic code execution from untrusted sources (`eval`, `exec`, dynamic imports)
- All target URLs are validated and scoped

### Safe Defaults

- TLS verification enabled by default
- Explicit timeouts on all HTTP requests
- No hardcoded credentials or API keys

## Input Validation

### Target URLs

- Must be explicitly configured (e.g. `TARGET_URL`, `--target`)
- Should be validated and normalised before use
- Future: allowlist of permitted hosts/domains

### Payloads

- Payloads are predefined or generated by controlled mutation
- No user-supplied payloads executed as code
- LLM-generated payloads (future) would be sandboxed and reviewed

## Dependencies

- **requests**: Used for HTTP with timeouts; no known critical vulnerabilities in current versions
- Regular `pip audit` and dependency updates recommended

## Reporting Security Issues

If you find a vulnerability in RMLW itself (not in targets it tests), please report responsibly. See [SECURITY.md](../SECURITY.md) in the project root.
